# 项目原始需求

## 目标

我想建立一个名为SRT Flow的Web工具，用于下载YouTube和Bilibili上的视频，并进行视频管理、语音识别、字幕翻译、语音合成、视频合成、简单剪辑，以及缩略图、标题、摘要等素材生成。  

## 功能

### 1. 视频下载

通过集成yt-dlp、BiliDown等工具，实现YouTube和Bilibili视频下载。  

功能优先级高。  

### 2. 视频管理：  

1. 生成并记录视频的唯一识别码，系统可以识别视频是否已经下载过，避免重复下载；  
2. 根据来源、用户、Channel（播放列表）、长度、大小等，给视频打上标签；  
3. 用户可以对视频自定义分组；  
4. 后续所有操作如语音识别、字幕翻译、语音合成、视频合成、简单剪辑均支持按组提交，排队流水线操作。  

功能优先级高。  

### 3. 语音识别

利用本地Whisper和在线大模型（如Gemini）实现语音识别，生成原始字幕。  

功能优先级高。  

### 4. 字幕翻译

利用在线大模型（如DeepSeek R1）实现字幕翻译，生成翻译字幕。  

功能优先级高。  

### 5. 语音合成

利用TTS技术将翻译后的字幕，合成语音。  

功能优先级中。  

### 6. 视频合成

利用FFMPEG将原始、翻译字幕按可配置的样式，叠加在原视频画面上，替换翻译后的语音（可选），并输出目标视频。  

功能优先级高。  

### 7. 简单剪辑

对于比较长的目标视频，可以进行简单预览、切分多段等操作。  

功能优先级低。  

### 8. 素材生成

系统可以根据视频字幕，利用大模型生成缩略图、标题、摘要等素材。  

功能优先级高。  

## 部署环境

Ubuntu 24.04及以上（支持Hyper-V或WSL2，支持Docker部署）  
Python 3.12 + Venv  
SQLite  

## 设计要求

### 1. 参数配置

所有功能的参数配置均存放在SQLite。敏感信息如API Key等需要加密存储，不允许在前端明文展示（前端只能修改，不能查询）。支持系统本身运行的常量如数据库文件名、文件存储路径等用dotEnv管理和读取。  
每个功能自身的配置，设计在该功能配置页面。整个项目共用的配置，放在系统配置内。  

### 2. 任务发布

所有的功能都要考虑到可能执行较长时间，因此倾向于前端发起任务之后，后端立即返回，然后前端轮询任务状态。  
为了简化操作，每个功能均支持把视频打包成任务组一次性提交。任务组本身也会成为视频标签，方便后续功能中选择。  

### 3. 用户管理

本系统定位为个人工具，因此不需要实现用户管理。  

### 4. 任务调度

考虑到资源和服务并发度，本系统每个功能只允许最多有一个任务在执行，如果有多个任务，需要排队。但是不同功能之间允许任务并行。  
类似于流水线操作。  
如果任务执行失败，则记录错误，自动调度队列中下一个任务。  
用户可以重启失败的任务，重启的时候可以指定任务排在队列最前还是最后。  

### 5. 任务时段

每个功能，均支持设置独立的允许任务执行的时段。当启动任务时，系统检查是否在允许执行的时段，如果不在时段则任务自动进入排队等待状态。  

### 6. 任务进度

系统支持任务进度的展示。  
系统可以主动将任务拆分为批次（如把一个很长的字幕文件拆分为30条一批进行翻译）并根据批次计算出进度，也可以被动的接收处理情况（如从FFMPEG读取处理字节数），进而计算出进度。  

### 7. 日志记录

由于需要调用本地或在线大模型，每个任务都要记录详细的日志，以便分析执行过程。  

### 8. 预览调整

所有任务的输出结果，系统要提供简单的预览和编辑界面。  

### 9. 结果输出

所有任务的结果，都要以标准的文件形式落地在文件系统，方便用户通过专业工具读取和处理。比如字幕用.srt文件，语音用.m4a文件，视频用.mp4等。  
所有的结果都要与原始视频文件关联。  
允许用户将自行处理后的结果文件关联到原始视频文件。  

### 10. 文件管理

每一个原始视频都对应一个目录，所有相关文件均保存在此目录。
