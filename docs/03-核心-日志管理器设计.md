# 日志管理器详细设计文档

## 1. 模块概述

### 1.1 功能定位

日志管理器是 SRT Flow 系统的核心基础模块，负责统一管理系统日志的记录、分流和存储。它为所有模块提供标准化的日志服务，支持按任务 ID 分流日志到独立文件，便于问题追踪和调试。

### 1.2 核心职责

- **统一日志接口**：为所有模块提供一致的日志记录 API
- **日志分级**：支持 DEBUG、INFO、WARNING、ERROR、CRITICAL 五个级别
- **任务日志分流**：按任务 ID 将日志写入独立文件（task.log）
- **日志格式化**：统一的日志格式，包含时间戳、级别、模块、消息
- **日志轮转**：自动管理日志文件大小和数量
- **控制台输出**：开发环境支持彩色控制台输出

### 1.3 设计原则

- **零配置启动**：使用合理默认值，无需额外配置即可工作
- **性能优先**：异步写入，避免阻塞业务逻辑
- **易于追踪**：每条日志包含足够的上下文信息
- **灵活扩展**：支持自定义 Handler 和 Formatter

## 2. 日志架构设计

### 2.1 日志流向

日志从各模块产生后，经过日志管理器统一处理，输出到多个目标：

1. **控制台**：开发调试用，支持彩色输出
2. **系统日志文件**（app.log）：记录所有日志，支持轮转
3. **错误日志文件**（error.log）：仅记录 ERROR 及以上级别
4. **任务日志文件**（task.log）：按视频目录分流，记录该视频所有任务日志
5. **数据库**（task_logs 表）：便于前端查询任务执行历史

### 2.2 日志文件结构

```
data/
├── logs/
│   ├── app.log              # 系统主日志（轮转）
│   ├── app.log.1            # 历史日志
│   ├── app.log.2
│   └── error.log            # 错误日志
└── downloads/
    └── <video_uuid>/
        └── task.log         # 该视频所有任务的日志
```

### 2.3 日志级别定义

| 级别 | 数值 | 用途 | 示例 |
|------|------|------|------|
| DEBUG | 10 | 调试信息，详细的执行过程 | 函数参数、中间变量 |
| INFO | 20 | 常规信息，关键业务节点 | 任务开始/完成、文件创建 |
| WARNING | 30 | 警告信息，不影响执行但需关注 | 重试、降级、配置缺失 |
| ERROR | 40 | 错误信息，功能受影响但可恢复 | API 调用失败、文件读取错误 |
| CRITICAL | 50 | 严重错误，系统无法继续 | 数据库连接失败、关键配置缺失 |

## 3. 日志格式设计

### 3.1 标准日志格式

```
[时间戳] [级别] [模块名] [任务ID] - 消息内容
```

示例：
```
[2024-12-15 10:30:45.123] [INFO    ] [download_service] [task-abc123] - Starting download
[2024-12-15 10:30:46.456] [ERROR   ] [asr_service] [task-def456] - Whisper model load failed
```

### 3.2 格式字段说明

| 字段 | 格式 | 说明 |
|------|------|------|
| 时间戳 | YYYY-MM-DD HH:MM:SS.mmm | 精确到毫秒 |
| 级别 | 8 字符左对齐 | DEBUG/INFO/WARNING/ERROR/CRITICAL |
| 模块名 | 变长 | Logger 名称，通常为模块路径 |
| 任务ID | 变长 | 当前任务 ID，无任务时显示 "-" |
| 消息 | 变长 | 日志内容 |

### 3.3 控制台彩色输出

开发环境使用彩色输出提升可读性：

| 级别 | 颜色 |
|------|------|
| DEBUG | 灰色 |
| INFO | 绿色 |
| WARNING | 黄色 |
| ERROR | 红色 |
| CRITICAL | 红色加粗 |

### 3.4 JSON 格式（可选）

用于日志聚合和分析场景，包含以下字段：
- timestamp：ISO 8601 格式时间戳
- level：日志级别
- logger：模块名
- task_id：任务 ID
- video_id：视频 ID
- message：日志消息
- extra：附加数据
- exception：异常信息（如有）

## 4. 任务日志分流方案

### 4.1 分流机制

每个视频目录下有独立的 task.log 文件，记录该视频所有任务的执行日志。分流基于任务上下文实现。

### 4.2 任务上下文

使用 Python contextvars 实现任务上下文的自动传递：

| 上下文变量 | 说明 |
|------------|------|
| current_task_id | 当前任务 ID |
| current_video_id | 当前视频 ID |
| current_video_path | 当前视频目录路径 |

### 4.3 上下文管理器

提供 TaskContext 上下文管理器，在任务执行时设置上下文：

1. 进入上下文时设置 task_id、video_id、video_path
2. 上下文内的所有日志自动携带这些信息
3. 日志同时写入系统日志和对应视频的 task.log
4. 退出上下文时自动清理

### 4.4 任务日志 Handler

TaskFileHandler 负责将日志写入视频目录：

1. 从上下文获取 video_path
2. 如果有 video_path，获取或创建对应的文件 Handler
3. 将日志写入 `{video_path}/task.log`
4. Handler 缓存复用，避免频繁打开文件

## 5. 日志管理器实现

### 5.1 单例模式

日志管理器采用单例模式，确保全局唯一实例和一致的配置。

### 5.2 初始化流程

1. 创建日志目录（如不存在）
2. 获取根 Logger
3. 清除已有 Handler
4. 配置控制台 Handler（可选彩色）
5. 配置主日志文件 Handler（带轮转）
6. 配置错误日志文件 Handler
7. 配置任务日志 Handler

### 5.3 配置选项

| 选项 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| log_dir | path | data/logs | 日志目录 |
| log_level | enum | INFO | 日志级别 |
| enable_console | bool | true | 启用控制台输出 |
| enable_color | bool | true | 启用彩色输出 |
| enable_json | bool | false | 启用 JSON 格式 |
| max_bytes | int | 10MB | 单文件最大大小 |
| backup_count | int | 5 | 保留的历史文件数 |

### 5.4 日志获取

提供 `get_logger(name)` 函数获取 Logger 实例，带缓存避免重复创建。

## 6. 日志文件管理

### 6.1 日志轮转配置

| 文件 | 最大大小 | 保留数量 | 说明 |
|------|----------|----------|------|
| app.log | 10MB | 5 | 主日志，轮转后为 app.log.1, app.log.2... |
| error.log | 10MB | 5 | 错误日志 |
| task.log | 不限制 | - | 每个视频独立，随视频删除 |

### 6.2 日志清理策略

- 系统日志：通过轮转自动管理，保留最近 5 个文件
- 任务日志：随视频目录一起管理，删除视频时一并删除
- 可配置保留天数，定期清理过期的轮转日志

### 6.3 日志查询

支持从 task.log 文件读取日志：
- 按级别过滤
- 分页查询
- 倒序返回（最新在前）

---

## 7. 数据库日志存储

### 7.1 存储策略

任务日志同时写入数据库 task_logs 表，便于前端查询：

- 使用缓冲区批量写入，减少数据库压力
- 缓冲区大小默认 100 条
- 定时刷新或缓冲区满时写入

### 7.2 查询接口

通过 API 查询任务日志：
- 按 task_id 查询
- 按级别过滤
- 分页支持
- 时间范围过滤

## 8. 使用指南

### 8.1 基础使用

1. 应用启动时初始化 LogManager
2. 在模块中通过 `get_logger(__name__)` 获取 Logger
3. 使用标准日志方法：debug/info/warning/error/critical

### 8.2 任务上下文使用

1. 任务开始时创建 TaskContext
2. 在 with 块内执行任务逻辑
3. 所有日志自动携带任务信息并分流到 task.log

### 8.3 日志级别使用建议

| 场景 | 推荐级别 |
|------|----------|
| 函数入参、中间变量 | DEBUG |
| 业务流程节点 | INFO |
| 可恢复的异常、重试 | WARNING |
| 功能失败 | ERROR |
| 系统级故障 | CRITICAL |

---

## 9. 错误处理

| 异常类型 | 说明 |
|----------|------|
| LogFileError | 日志文件操作错误 |
| LogConfigError | 日志配置错误 |

日志系统本身的错误不应影响业务，采用降级策略：
- 文件写入失败时降级到控制台
- 数据库写入失败时仅记录到文件
