# 流程管理（Pipeline）功能设计

## 1. 功能概述

用户在视频管理页选择已下载的视频，配置需要顺次执行的处理步骤
（如：语音识别 → 字幕翻译 → 语音合成 → 视频合成），点击开始后，
系统在后端自动串联执行，直到所有步骤完成。

**核心约束：**

- 单视频，V1 不支持批量
- 步骤参数使用全局默认配置，不单独设置
- 后端流水线驱动，关闭浏览器不影响执行
- 不跳过步骤，即使产出文件已存在（覆盖执行）
- 系统重启后从中断点恢复
- 步骤失败自动退避重试，超出上限后流程标记失败

---

## 2. 现有基础分析

| 组件 | 现状 | 本次变动 |
|------|------|----------|
| `PipelineOrchestrator` | 存在，静态 `PIPELINE_RULES` + 布尔开关 | 新增动态路由分支，静态规则保留兼容 |
| `QueueManager.recover_interrupted_tasks()` | 已有，重启后 running→pending | 启动时配合 `PipelineService.recover_pipelines()` 使用 |
| `Task.payload` | JSON 字段 | 写入 `pipeline_id`、`pipeline_steps`、`pipeline_step_index`，无 schema 变更 |
| `TaskEvent.COMPLETED / FAILED` | 已有事件系统 | `PipelineService` 订阅，处理步骤推进和失败 |
| `RetryScheduler` | 指数退避重试已有 | 直接复用，Pipeline 只关注「超出上限」的最终失败 |
| 数据库 | 无 Pipeline 表 | **新增 `pipelines` 表**，持久化编排状态 |

## 3. 数据模型

### 3.1 新增 `Pipeline` 表

```python
class PipelineStatus(str, Enum):
    RUNNING   = "running"    # 正在执行
    COMPLETED = "completed"  # 全部步骤完成
    FAILED    = "failed"     # 某步骤超出重试上限
    CANCELLED = "cancelled"  # 用户主动取消

class Pipeline(Base):
    __tablename__ = "pipelines"

    id:              str             # UUID 主键
    video_id:        str             # 关联视频（FK → videos.id CASCADE DELETE）
    steps:           list[str]       # 步骤列表，如 ["asr","translate","tts","synthesize"]
    current_step:    int             # 当前步骤索引（0-based），初始为 0
    status:          PipelineStatus  # 流程整体状态
    current_task_id: str | None      # 当前正在执行的 Task ID
    error:           str | None      # 最后的错误信息
    created_at:      datetime
    updated_at:      datetime
```

### 3.2 Task.payload 新增字段（无 schema 变更）

所有 Pipeline 产生的任务，payload 中额外携带：

```json
{
  "pipeline_id": "uuid",
  "pipeline_steps": ["asr", "translate", "tts", "synthesize"],
  "pipeline_step_index": 0
}
```

调度器通过 `pipeline_id` 字段是否存在来区分「Pipeline 任务」和「独立任务」，
Pipeline 任务的完成事件交由 `PipelineService` 处理，不走 `PIPELINE_RULES` 静态规则。

## 4. 步骤依赖规则

支持步骤及依赖关系：

```
asr
 └─ translate
      ├─ tts
      │   └─ synthesize
      └─ asset_gen
```

| 步骤 | 中文名 | 前置依赖 |
|------|--------|----------|
| asr | 语音识别 | 无 |
| translate | 字幕翻译 | asr |
| tts | 语音合成 | translate |
| synthesize | 视频合成 | tts |
| asset_gen | 素材生成 | translate |

**校验规则：**

- 前端选择步骤时实时提示缺失的前置步骤（不自动勾选，由用户确认）
- 后端 `POST /api/v1/pipelines` 创建时二次校验，不合法返回 400

### 5.2 新增文件

**`backend/services/pipeline_service.py`**

```python
class PipelineService:

    async def create_pipeline(video_id: str, steps: list[str]) -> Pipeline:
        """
        1. 校验步骤依赖合法性
        2. 校验视频存在
        3. 创建 Pipeline 记录（status=running, current_step=0）
        4. 创建第一个 Task，payload 写入 pipeline 上下文
        5. 更新 Pipeline.current_task_id
        """

    async def on_task_completed(task: Task) -> None:
        """
        任务完成时由调度器回调。
        1. 读取 task.payload["pipeline_id"]，加载 Pipeline
        2. 推进 current_step += 1
        3. 若还有下一步：创建下一个 Task，更新 current_task_id
        4. 若已是最后步骤：Pipeline.status = completed
        5. 推送 WebSocket pipeline_update 事件
        """

    async def on_task_failed(task: Task) -> None:
        """
        任务失败时由调度器回调。
        1. 读取 pipeline_id，加载 Pipeline
        2. 若 task.retry_count < task.max_retries：不处理，RetryScheduler 负责重试
        3. 若 task.retry_count >= task.max_retries：
           Pipeline.status = failed，error = task.error
           推送 WebSocket pipeline_update 事件
        """

    async def cancel_pipeline(pipeline_id: str) -> Pipeline:
        """
        1. 加载 Pipeline，校验状态为 running
        2. 取消 current_task_id 对应的 Task（pending→cancelled，running→cancelled）
        3. Pipeline.status = cancelled
        """

    async def recover_pipelines() -> None:
        """
        系统启动时调用（在 QueueManager.recover_interrupted_tasks() 之后）。
        查询所有 status=running 的 Pipeline：
        - current_task_id 对应 Task 为 completed 但 pipeline 未推进
          → 补触发 on_task_completed（处理上次完成后宕机的情况）
        - current_task_id 对应 Task 为 pending/running
          → 不处理（已由 QueueManager 恢复，等待队列正常执行）
        - current_task_id 对应 Task 为 failed 且 retry_count >= max_retries
          → Pipeline.status = failed
        """
```

### 5.3 改造文件

**`backend/workers/scheduler.py` — `PipelineOrchestrator`**

在 `_on_task_completed` 和 `_on_task_failed` 中：

```python
def _on_task_completed(self, event, task_id, data):
    asyncio.create_task(self._dispatch_completed(task_id))

async def _dispatch_completed(self, task_id):
    task = await self._queue.get_task(task_id)
    if task.payload and task.payload.get("pipeline_id"):
        # Dynamic pipeline: delegate to PipelineService
        await pipeline_service.on_task_completed(task)
    else:
        # Static pipeline: use PIPELINE_RULES (existing logic)
        await self._trigger_downstream(task)
```

**`backend/app/main.py` — lifespan 启动序列**

```python
# 已有
await queue_manager.recover_interrupted_tasks()
# 新增（在其后）
await pipeline_service.recover_pipelines()
```

### 5.4 退避重试机制

完全复用现有 `RetryScheduler`，配置参数：

| 参数 | 值 |
|------|-----|
| 初始等待 | 60 秒 |
| 退避系数 | 2.0（指数退避） |
| 上限等待 | 3600 秒（1 小时） |
| 最大重试次数 | 3 次（可由 Task.max_retries 配置） |

重试延迟序列：60s → 120s → 240s → 失败

Pipeline 层不干预重试过程，仅在 `on_task_failed` 检测到
`retry_count >= max_retries` 时将流程标记为 `failed`。

## 6. WebSocket 推送

复用 `/ws/status`，新增事件类型 `pipeline_update`：

```json
{
  "type": "pipeline_update",
  "data": {
    "pipeline_id": "uuid",
    "video_id": "uuid",
    "status": "running",
    "steps": ["asr", "translate", "tts", "synthesize"],
    "current_step": 2,
    "current_task_id": "uuid",
    "error": null
  }
}
```

触发时机：

- Pipeline 创建（步骤 0 启动）
- 每个步骤完成、下一步启动
- 流程 completed / failed / cancelled

## 9. 状态流转图

```
用户点击「开始」
       ↓
POST /api/v1/pipelines
       ↓
Pipeline: running, step=0
Task[asr]: pending
       ↓
Task[asr]: running → completed
       ↓
PipelineService.on_task_completed()
Pipeline: running, step=1
Task[translate]: pending
       ↓
  ... （以此类推）
       ↓
Task[最后步骤]: completed
       ↓
Pipeline: completed ✓

── 任意步骤失败 ──
Task: failed (retry_count < max)
       ↓
RetryScheduler 指数退避重试
（60s → 120s → 240s → ...，上限 3600s）
Pipeline 保持 running，等待重试结果
       ↓
重试成功 → 继续下一步
重试耗尽 → Pipeline: failed ✗

── 系统重启 ──
QueueManager.recover_interrupted_tasks()
  running Task → pending（自动重试）
PipelineService.recover_pipelines()
  检查 running Pipeline 的 current_task_id
  Task 状态正常 → 不处理，等待队列执行
  Task completed 但 pipeline 未推进 → 补触发下一步
  Task failed 且耗尽重试 → Pipeline: failed ✗

── 用户取消 ──
POST /api/v1/pipelines/{id}/cancel
  当前 pending Task → cancelled
  当前 running Task → cancelled（发信号中止）
  Pipeline: cancelled ✗
```
