# 调度器详细设计文档

## 1. 模块概述

### 1.1 功能定位

调度器（Scheduler）是 SRT Flow 任务调度系统的控制中心，负责任务的定时检查、错误重试、时段控制和流水线编排。它与队列管理器和工作进程协同工作，确保任务按预期执行。

### 1.2 核心职责

- **定时检查**：周期性检查任务状态，处理异常情况
- **错误重试**：管理失败任务的重试调度
- **时段控制**：根据配置的时间段控制任务执行
- **流水线编排**：任务完成后自动触发下游任务
- **超时处理**：检测并处理执行超时的任务
- **系统恢复**：启动时恢复中断的任务

### 1.3 设计原则

- **可靠性**：确保任务不丢失、不重复执行
- **灵活性**：支持多种调度策略和配置
- **可观测**：提供调度状态的监控和查询
- **低侵入**：与业务逻辑解耦

## 3. 错误重试策略

### 3.1 重试规则

| 规则 | 说明 |
|------|------|
| 最大重试次数 | 默认 3 次，可按任务类型配置 |
| 重试间隔 | 指数退避：1分钟、5分钟、15分钟 |
| 可重试错误 | 网络超时、API 限流、临时服务不可用 |
| 不可重试错误 | 参数错误、认证失败、文件不存在 |

### 3.2 重试流程

1. 任务执行失败，Worker 更新状态为 `failed`
2. 调度器检测到失败任务
3. 检查 `retry_count` 是否小于 `max_retries`
4. 检查错误类型是否可重试
5. 计算下次重试时间（指数退避）
6. 设置 `scheduled_at` 并将状态改为 `pending`
7. 到达重试时间后，Worker 获取并执行

### 3.3 指数退避算法

重试间隔计算公式：`base_interval * (2 ^ retry_count)`

| 重试次数 | 间隔（默认配置） |
|----------|------------------|
| 第 1 次 | 1 分钟 |
| 第 2 次 | 2 分钟 |
| 第 3 次 | 4 分钟 |

可配置最大间隔上限，避免等待时间过长。

### 3.4 重试配置

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| max_retries | int | 3 | 最大重试次数 |
| base_retry_interval | int | 60 | 基础重试间隔（秒） |
| max_retry_interval | int | 3600 | 最大重试间隔（秒） |
| retry_backoff_factor | float | 2.0 | 退避因子 |

## 4. 任务时段控制

### 4.1 功能说明

每种任务类型可配置允许执行的时间段，不在时段内的任务自动延迟到下一个允许时段。

### 4.2 时段配置

每种任务类型支持配置：
- `allowed_time_start`：允许开始时间（HH:MM 格式）
- `allowed_time_end`：允许结束时间（HH:MM 格式）

示例配置：
| 任务类型 | 开始时间 | 结束时间 | 说明 |
|----------|----------|----------|------|
| download | 00:00 | 08:00 | 夜间下载，避免占用带宽 |
| asr | null | null | 不限制 |
| tts | 22:00 | 06:00 | 夜间执行，避免 GPU 资源竞争 |

### 4.3 时段检查流程

1. Worker 尝试获取任务
2. 调度器检查当前时间是否在允许时段
3. 如果在时段内，正常执行
4. 如果不在时段内：
   - 计算下一个允许时段的开始时间
   - 设置任务的 `scheduled_at`
   - 任务保持 `pending` 状态等待

### 4.4 跨日时段处理

支持跨日时段配置（如 22:00 - 06:00）：
- 开始时间 > 结束时间 表示跨日
- 检查逻辑：当前时间 >= 开始时间 OR 当前时间 < 结束时间

## 5. 流水线编排

### 5.1 流水线定义

视频处理的标准流水线：

```
download → asr → translate → tts → synthesize
                     ↓
                 asset_gen
```

### 5.2 触发规则

| 上游任务 | 下游任务 | 触发条件 |
|----------|----------|----------|
| download | asr | 下载完成且启用 ASR |
| asr | translate | ASR 完成且启用翻译 |
| translate | tts | 翻译完成且启用 TTS |
| translate | asset_gen | 翻译完成且启用素材生成 |
| tts | synthesize | TTS 完成且启用视频合成 |

### 5.3 流水线配置

用户提交任务时可选择：
- **单步执行**：只执行指定任务
- **流水线执行**：自动执行后续所有任务
- **部分流水线**：指定执行到哪一步

### 5.4 流水线状态跟踪

为流水线任务组维护整体状态：
- 记录流水线包含的所有任务
- 跟踪整体进度（已完成步骤/总步骤）
- 任一步骤失败时标记流水线失败

## 6. 超时处理

### 6.1 超时检测

定期扫描 `running` 状态的任务，检查执行时间是否超过阈值。

### 6.2 超时配置

| 任务类型 | 默认超时 | 说明 |
|----------|----------|------|
| download | 2 小时 | 大文件下载可能较慢 |
| asr | 1 小时 | 取决于视频长度和模型 |
| translate | 30 分钟 | API 调用通常较快 |
| tts | 1 小时 | 取决于文本长度 |
| synthesize | 1 小时 | 取决于视频长度 |
| asset_gen | 30 分钟 | API 调用 |

### 6.3 超时处理流程

1. 检测到任务执行超时
2. 尝试通知 Worker 取消任务
3. 等待 Worker 响应（30 秒）
4. 如果 Worker 无响应，强制标记任务为 `failed`
5. 记录超时错误信息
6. 根据重试策略决定是否重试

## 7. 系统恢复

### 7.1 恢复场景

系统重启时需要处理以下情况：
- `running` 状态的任务（被中断）
- `pending` 状态但 `scheduled_at` 已过期的任务

### 7.2 恢复策略

| 任务类型 | 恢复策略 | 说明 |
|----------|----------|------|
| download | 重新开始 | 下载工具支持断点续传 |
| asr | 重新开始 | 无法恢复中间状态 |
| translate | 重新开始 | 批次处理，重新开始更简单 |
| tts | 重新开始 | 无法恢复中间状态 |
| synthesize | 重新开始 | FFMPEG 不支持恢复 |

### 7.3 恢复流程

1. 系统启动时执行恢复检查
2. 查询所有 `running` 状态的任务
3. 根据任务类型和恢复策略处理：
   - 可恢复：重置为 `pending`，`retry_count` 不变
   - 不可恢复：标记为 `failed`
4. 检查过期的 `scheduled_at`，清除延迟标记
5. 记录恢复日志

## 8. 定时任务

### 8.1 调度器定时任务列表

| 任务 | 间隔 | 说明 |
|------|------|------|
| 重试检查 | 30 秒 | 检查需要重试的失败任务 |
| 超时检查 | 1 分钟 | 检查执行超时的任务 |
| 时段检查 | 1 分钟 | 检查时段限制的任务 |
| 健康检查 | 5 分钟 | 检查系统整体健康状态 |
| 清理任务 | 1 小时 | 清理过期的已完成任务记录 |

### 8.2 定时任务实现

使用异步定时器实现，避免阻塞主流程：
- 每个定时任务独立运行
- 任务执行时间超过间隔时跳过本次
- 支持手动触发

## 9. 事件处理

### 9.1 监听的事件

| 事件 | 处理逻辑 |
|------|----------|
| task.completed | 触发流水线下游任务 |
| task.failed | 评估是否需要重试 |
| worker.error | 检查是否需要任务恢复 |
| config.updated | 重新加载调度配置 |

### 9.2 发出的事件

| 事件 | 触发时机 |
|------|----------|
| scheduler.retry_scheduled | 安排了重试任务 |
| scheduler.task_timeout | 检测到任务超时 |
| scheduler.pipeline_triggered | 触发了流水线任务 |
| scheduler.recovery_completed | 系统恢复完成 |
