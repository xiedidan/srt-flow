# SRT Flow 项目需求文档

## 1. 项目概述

SRT Flow 是一个基于 Web 的视频处理工具，用于下载 YouTube 和 Bilibili 上的视频，并进行视频管理、语音识别、字幕翻译、语音合成、视频合成、简单剪辑，以及缩略图、标题、摘要等素材生成。项目定位为个人工具，强调高质量的输出和现代化的用户体验。

## 2. 技术架构

- **前端框架**：Vue.js (单页应用)
- **后端框架**：FastAPI (Python 3.12)
- **数据库**：SQLite (用于存储任务状态、视频元数据、配置信息)
- **任务队列**：持久化队列 (基于 SQLite)，支持服务重启后任务不丢失
- **部署方式**：Docker / Docker Compose
- **运行环境**：Ubuntu 24.04+，支持 Hyper-V 或 WSL2，支持 GPU 加速 (CUDA)

## 3. 功能需求

### 3.1 视频下载（优先级：高）

- **支持平台**：YouTube, Bilibili
- **下载工具**：集成 yt-dlp、BiliDown 等工具
- **核心功能**：
  - 输入 URL 下载视频
  - 仅支持公开视频（无需登录/Cookie）
  - 自动提取视频元数据（标题、时长、上传者、上传日期等）

### 3.2 视频管理（优先级：高）

- **唯一识别码**：
  - 生成并记录视频的唯一识别码
  - 系统可以识别视频是否已经下载过，避免重复下载
  
- **自动标签系统**：
  - 根据来源（YouTube/Bilibili）自动打标签
  - 根据用户（上传者）自动打标签
  - 根据 Channel（播放列表）自动打标签
  - 根据长度、大小等属性自动打标签
  
- **用户自定义分组**：
  - 用户可以对视频进行自定义分组
  - 分组可用于批量操作
  
- **批量操作支持**：
  - 后续所有操作（语音识别、字幕翻译、语音合成、视频合成、简单剪辑）均支持按组提交
  - 支持排队流水线操作

### 3.3 语音识别（优先级：高）

- **本地引擎**：OpenAI Whisper
  - 默认使用 `large-v3` 模型（高精度）
  - 支持用户在界面配置模型大小
  
- **在线引擎**：支持调用在线大模型（如 Gemini）
  - 作为备选方案
  - 可在配置中切换
  
- **输出**：生成原始字幕文件（.srt 格式）

### 3.4 字幕翻译（优先级：高）

- **引擎**：在线大模型（如 DeepSeek R1, Gemini, OpenAI）
- **功能**：
  - 读取原始字幕文件
  - 生成翻译字幕（.srt 格式）
  - 支持批次处理（如每 30 条字幕为一批）以优化上下文和进度计算

### 3.5 语音合成（优先级：中）

- **引擎**：本地 TTS（Coqui TTS / ChatTTS / SparkTTS / IndexTTS / CozyVoice / VITS系）
- **功能**：
  - 将翻译后的字幕合成语音
  - 支持根据字幕长度和字数计算语速，探测合成语音速度
  - 支持 GPU 加速
  - 输出音频文件（.wav 或 .m4a 格式，可配置）

### 3.6 视频合成（优先级：高）

- **工具**：FFMPEG
- **功能**：
  - 将原始字幕、翻译字幕按可配置的样式叠加在原视频画面上
  - 替换翻译后的语音（可选）：完全静音原视频，使用 TTS 生成的语音
  - 输出目标视频（.mp4 格式）
  
- **样式配置**：
  - 基础：字体、大小、颜色、位置
  - 进阶：边框、阴影、背景色、动画效果

### 3.7 简单剪辑（优先级：低）

- **功能**：
  - 对于比较长的目标视频，可以进行简单预览
  - 支持切分多段等操作

### 3.8 素材生成（优先级：高）

- **功能**：根据视频字幕，利用大模型生成素材
  
- **缩略图生成**：
  - 选项 A：截取视频帧
  - 选项 B：LLM 生成提示词 + 调用绘图接口（绘图接口可配置）
  - 用户可在界面选择使用哪种方式
  
- **其他素材**：
  - 标题生成
  - 摘要生成

## 4. 系统设计要求

### 4.1 参数配置

- **存储位置**：
  - 所有功能的参数配置均存放在 SQLite
  - 系统运行常量（如数据库文件名、文件存储路径等）用 .env 管理和读取
  
- **安全性**：
  - 敏感信息（如 API Key）需要加密存储
  - 前端不允许明文展示（前端只能修改，不能查询）
  
- **配置组织**：
  - 每个功能自身的配置，设计在该功能配置页面
  - 整个项目共用的配置，放在系统配置内

### 4.2 任务发布

- **异步处理**：
  - 所有的功能都要考虑到可能执行较长时间
  - 前端发起任务之后，后端立即返回
  - 前端轮询任务状态
  
- **任务组**：
  - 每个功能均支持把视频打包成任务组一次性提交
  - 任务组本身也会成为视频标签，方便后续功能中选择

### 4.3 用户管理

- **定位**：本系统定位为个人工具
- **实现**：不需要实现用户管理

### 4.4 任务调度

- **并发控制**：
  - 每个功能只允许最多有一个任务在执行
  - 如果有多个任务，需要排队
  - 不同功能之间允许任务并行（类似于流水线操作）
  
- **错误处理**：
  - 如果任务执行失败，则记录错误
  - 自动调度队列中下一个任务
  
- **任务重启**：
  - 用户可以重启失败的任务
  - 重启时可以指定任务排在队列最前还是最后

### 4.5 任务时段

- **功能**：每个功能均支持设置独立的允许任务执行的时段
- **逻辑**：
  - 当启动任务时，系统检查是否在允许执行的时段
  - 如果不在时段则任务自动进入排队等待状态

### 4.6 任务进度

- **展示**：系统支持任务进度的展示
  
- **计算方式**：
  - 主动拆分：系统可以主动将任务拆分为批次（如把一个很长的字幕文件拆分为 30 条一批进行翻译）并根据批次计算出进度
  - 被动接收：可以被动的接收处理情况（如从 FFMPEG 读取处理字节数），进而计算出进度

### 4.7 日志记录

- **需求**：由于需要调用本地或在线大模型，每个任务都要记录详细的日志
- **目的**：以便分析执行过程

### 4.8 预览调整

- **功能**：所有任务的输出结果，系统要提供简单的预览和编辑界面
- **示例**：
  - 字幕编辑器
  - 视频预览播放器

### 4.9 结果输出

- **文件格式**：
  - 所有任务的结果，都要以标准的文件形式落地在文件系统
  - 字幕用 .srt 文件
  - 语音用 .m4a 文件
  - 视频用 .mp4 等
  
- **文件关联**：
  - 所有的结果都要与原始视频文件关联
  - 允许用户将自行处理后的结果文件关联到原始视频文件

### 4.10 文件管理

- **目录结构**：每一个原始视频都对应一个目录，所有相关文件均保存在此目录
- **示例**：
  ```
  <Video_UUID>_<Safe_Title>/
  ├── video.mp4                    # 原始视频
  ├── video.info.json              # 元数据
  ├── audio_original.m4a           # 提取的原声音频
  ├── subtitle_original.srt        # 识别出的源语言字幕
  ├── subtitle_translated.srt      # 翻译后的目标语言字幕
  ├── audio_tts.m4a                # 合成的语音
  ├── video_output.mp4             # 最终合成视频
  ├── assets/                      # 素材子目录
  │   ├── thumbnail.jpg
  │   ├── summary.txt
  │   └── title_candidates.txt
  └── task.log                     # 该视频处理过程的日志
  ```

## 5. 用户界面要求

- **风格**：现代、科技感、深色模式
- **交互**：
  - 任务进度实时展示（轮询或 WebSocket）
  - 结果预览与编辑（字幕编辑、视频预览）
  - 配置管理界面（每个功能独立配置页面 + 系统全局配置）

## 6. 部署要求

- **服务端口**：8010
- **交付物**：
  - 源代码
  - `docker-compose.yml`
  - 安装与使用文档
